## Package Organization

TheRock generates packages organized into several categories:

### Artifact Types
- **base**: Core ROCm infrastructure and utilities
- **amd-llvm**: Compiler toolchain
- **core-runtime**: Low-level runtime components
- **core-hip**: HIP runtime and development tools
- **hipify**: CUDA-to-HIP conversion tools
- **rocprofiler-sdk**: Profiler tools and libraries
- **prim**: Primitive libraries (rocPRIM, hipCUB, rocThrust)
- **blas**: Linear algebra libraries
- **fft**: FFT libraries
- **rand**: Random number generation libraries
- **rccl**: Communication collectives
- **miopen**: Machine learning primitives
- **composable-kernel**: High-performance composable kernels
- **support**: Development support headers
- **sysdeps**: Bundled system dependencies
- **host-blas**: Host BLAS libraries
- **host-suite-sparse**: SuiteSparse libraries

### Package Components
Each package can contain multiple components:
- **lib**: Runtime libraries (.so files, DLLs)
- **dev**: Development files (headers, CMake configs, static libraries)
- **run**: Executable binaries and tools
- **doc**: Documentation
- **dbg**: Debug symbols

### Standard CMake Config Structure

Most packages follow the standard CMake package configuration structure:

## Artifact Components

TheRock uses a component-based artifact system:

### Component Types

1. **lib** (Library Component)
   - Shared libraries (.so, .dll)
   - Runtime dependencies
   - Plugin directories
   - SONAME symlinks

2. **dev** (Development Component)
   - **CMake configuration files** (primary location)
   - Header files (.h, .hpp)
   - Static libraries (.a)
   - Import libraries (.lib on Windows)
   - pkg-config files
   - Module files

3. **run** (Runtime Component)
   - Executable binaries
   - Command-line tools
   - Scripts

4. **doc** (Documentation Component)
   - Documentation files
   - README files
   - License files
   - Sample code

5. **dbg** (Debug Component)
   - Debug symbol files (.debug, .pdb)
   - Source maps

6. **test** (Testing Component)
   - Test binaries
   - Test data
   - Test configuration files

## Package Dependencies

### Dependency Hierarchy

```
rocm-core (base)
  ├── rocm-cmake
  ├── half
  ├── rocm-smi-lib
  └── rocprofiler-register

amd-llvm (compiler)
  ├── Depends: rocm-core
  └── Provides: rocm-llvm, comgr, hipcc

hsa-rocr (runtime)
  ├── Depends: rocm-core, rocprofiler-register
  └── Required by: hip-runtime-amd

hip-runtime-amd
  ├── Depends: hsa-rocr, rocminfo, comgr, rocprofiler-register
  └── Required by: All GPU libraries

hip-devel
  ├── Depends: hip-runtime-amd, rocm-llvm, hsa-rocr-devel, hipcc
  └── Required by: Building HIP applications

rocblas
  ├── Depends: hip-runtime-amd, hipblaslt
  └── Required by: hipblas, rocsolver, miopen-hip

miopen-hip
  ├── Depends: hip-runtime-amd, comgr, roctracer, rocblas, hipblaslt, rocrand
  └── Optionally depends on: composable-kernel

rccl (Linux only)
  ├── Depends: hip-runtime-amd, rocm-smi-lib
  └── For multi-GPU communication
```

### Platform-Specific Packages

#### Linux Only
- hsa-rocr / hsa-rocr-devel
- rocminfo
- rccl / rccl-devel
- hipDNN
- miopen-plugin
- rocm-host-suite-sparse

#### Windows Only
- hipInfo (alternative to rocminfo)

#### Target-Neutral (Generic)
All packages in the "base" and "compiler" categories are target-neutral and work across different GPU architectures.

#### GPU-Specific (gfx*)
All math libraries (BLAS, FFT, RAND, PRIM), ML libraries, and RCCL are built per GPU target family (e.g., gfx110X-dgpu, gfx94X-dcgpu).
